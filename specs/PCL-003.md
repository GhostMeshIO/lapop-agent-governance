## 1. Purpose

PCL-003 defines the deterministic policy composition layer for multi-agent LLM orchestration.

It compiles policy fragments from multiple sources into a single effective policy set for a specific execution context.

PCL-003 answers the question:

> Given this context, agent, artifact, action, policy stack, and exception state, is this action allowed?

PCL exists because multi-agent systems frequently operate under overlapping constraints:

* user instructions,
* project rules,
* organization rules,
* artifact boundaries,
* safety policies,
* domain policies,
* tool restrictions,
* temporary overrides,
* human approvals,
* and emergency exceptions.

Without a deterministic policy composition layer, agents may follow whichever rule is most recent, most convenient, or most loudly repeated.

PCL prevents that.

---

## 2. Design Goals

PCL-003 aims to make policy enforcement:

* deterministic,
* auditable,
* context-aware,
* precedence-driven,
* conflict-detecting,
* exception-bounded,
* rollback-compatible,
* and reproducible.

The central invariant:

> Policy decisions must be explainable after the fact.

Every policy decision should be traceable to:

* active policy fragments,
* context,
* scope,
* precedence rules,
* conflict resolution,
* exception windows,
* and final allowance result.

---

## 3. Non-Goals

PCL-003 does **not** define:

* worker execution,
* artifact validation internals,
* trust graph internals,
* strategy learning internals,
* economic value-of-information algorithms,
* temporal rollback mechanics,
* prompt templates,
* or model behavior.

PCL only answers policy questions.

It does not execute actions.

It does not merge artifacts.

It does not learn from outcomes.

It does not decide whether work is economically worthwhile.

It does not rewrite the task.

---

## 4. Core Responsibilities

| Responsibility            | Description                                                          |
| ------------------------- | -------------------------------------------------------------------- |
| Policy Fragment Ingestion | Load policy fragments from multiple sources.                         |
| Scope Resolution          | Determine which policies apply to a context.                         |
| Precedence Composition    | Apply deterministic precedence rules.                                |
| Conflict Detection        | Detect contradictory allow/deny/require rules.                       |
| Conflict Resolution       | Resolve conflicts according to precedence or block.                  |
| Exception Management      | Allow bounded temporary exceptions.                                  |
| Policy Attestation        | Seal policy state for reproducibility.                               |
| Runtime Allowance         | Answer `is_action_allowed` queries.                                  |
| Constraint Emission       | Return required constraints when an action is allowed conditionally. |
| Audit Support             | Explain why a decision was made.                                     |
| Rollback Support          | Restore prior effective policy state by attestation ID.              |

## 6. Policy Precedence

PCL must apply deterministic precedence.

Recommended default order from strongest to weakest:

```yaml id="dpmwuj"
PolicyPrecedence:
  1: system_policy
  2: emergency_lockdown
  3: safety_policy
  4: organization_policy
  5: domain_policy
  6: project_policy
  7: repository_policy
  8: artifact_boundary_policy
  9: coordination_policy
  10: tool_policy
  11: human_override
  12: temporary_exception
```

### Important Rule

A lower-priority policy cannot override a higher-priority deny unless the higher-priority policy explicitly permits exception.

Example:

```text id="2i3f3b"
system_policy denies external publishing
temporary_exception allows external publishing
→ final result: denied
```

### Human Override Rule

Human override is powerful but not absolute.

It may override project, repository, coordination, and tool policies.

It may not override system, safety, or emergency lockdown policies unless explicitly allowed by the governing deployment.

### Decision Meaning

| Decision                       | Meaning                                           |

| `allow`                        | Action is allowed.                                |
| `allow_with_constraints`       | Action allowed only if constraints are satisfied. |
| `deny`                         | Action is forbidden.                              |
| `require_review`               | Supervisor or auditor review required.            |
| `require_dependency_note`      | CDO dependency note required.                     |
| `require_policy_exception`     | Formal exception required before action.          |

| `block_due_to_conflict`        | Policy conflict cannot be resolved safely.        |
| `block_due_to_missing_context` | More context needed before decision.              |

## 9. Policy Composition Flow

### 9.1 Input

PCL receives:

* execution context,
* action descriptor,
* policy fragments,
* active exceptions,
* artifact boundary constraints,
* environment metadata.

### 9.2 Scope Matching

PCL filters rules whose scope matches the context.

```python id="m70xyl"
def scope_matches(scope, context, action):
    return all([
        scope.project_id is None or scope.project_id == context.project_id,
        scope.artifact_id is None or scope.artifact_id == context.artifact_id,
        scope.agent_class is None or scope.agent_class == context.agent_class,
        scope.action_type is None or scope.action_type == action.action_type,
        scope.side_effect_class is None or scope.side_effect_class == action.side_effect_class,
    ])
```

### 9.3 Condition Evaluation

Rules with conditions are evaluated.

If context is missing required condition data, PCL may return:

```yaml id="dopy4a"
decision: block_due_to_missing_context
```

### 9.4 Conflict Detection

If active rules produce contradictory outcomes, PCL creates `PolicyConflict`.

Common examples:

```text id="xwhqsj"
allow write_internal vs deny artifact modification
allow tool call vs prohibit external write
human override vs emergency lockdown
temporary exception vs expired exception
```

### 9.5 Conflict Resolution

Conflicts resolve by:

1. explicit precedence,
2. stricter rule wins,
3. deny by default,
4. supervisor or human review,
5. exception window if permitted.

### 9.6 Attestation

PCL seals the effective policy state into `PolicyAttestationRecord`.

### 9.7 Allowance Result

PCL returns `AllowanceResult`.

## 11. Exception Windows

Exceptions are scoped and temporary.

They must include:

* who approved them,
* what actions they allow,
* what scope they apply to,
* when they expire,
* how many times they can be used,
* what constraints still apply,
* whether they are revocable.

### Exception Invariants

1. Exceptions expire.
2. Exceptions are scoped.
3. Exceptions are logged.
4. Exceptions cannot override non-exceptionable rules.
5. Exceptions must appear in policy attestation.
6. Exceptions must be rollback-aware.

---

## 12. Default Deny Cases

PCL should deny by default when:

* no policy applies to a high-risk action,
* policy conflict is unresolved,
* required context is missing,
* requested side effect is destructive,
* external write lacks explicit approval,
* artifact boundary is absent,
* policy exception is expired,
* contamination status is unknown where clean status is required,
* action would bypass validation,
* action would bypass dependency notes.

## 23. Conflict Handling

### 23.1 Allow vs Deny

Default:

```text id="zw9odi"
higher precedence wins
if equal precedence, deny by default
```

### 23.2 Incompatible Constraints

Default:

```text id="lx2q3e"
apply stricter compatible subset
if impossible, block_due_to_conflict
```

### 23.3 Ambiguous Precedence

Default:

```text id="3g2nc8"
block_due_to_conflict
```

### 23.4 Expired Exception

Default:

```text id="lg9t3x"
ignore exception
if action depends on exception, deny
```

### 23.5 Missing Authority

Default:

```text id="49g00l"
require_human_approval or deny
```

## 24. Safety Invariants

1. Deny beats allow unless precedence or valid exception says otherwise.
2. Higher-priority policy cannot be overridden by lower-priority policy.
3. Expired exceptions do not apply.
4. Unknown context blocks high-risk actions.
5. External writes require explicit approval.
6. Destructive actions require high-authority approval.
7. Contaminated runs cannot update strategy or validated memory.
8. Policy attestation must be recorded for significant actions.
9. Policy decisions must be explainable.
10. Policy restoration must be possible for rollback.

## 25. Failure Modes

| Failure Mode                | Description                        | Default Response                     |
| --------------------------- | ---------------------------------- | ------------------------------------ |
| No applicable policy        | No rule found for action           | Allow low-risk read; block high-risk |
| Policy conflict             | Active rules disagree              | Resolve by precedence or block       |
| Expired exception           | Exception no longer valid          | Ignore exception                     |
| Missing context             | Required field unavailable         | Block or request context             |
| Override abuse              | Too many exceptions                | CHL/EIL review                       |
| Policy drift                | Policy changes without attestation | Block and audit                      |
| Compiler mismatch           | Different compiler version         | Require reattestation                |
| Policy rollback failure     | Cannot restore prior policy state  | Escalate                             |
| Scope mismatch              | Rule applied to wrong scope        | Reject decision and recompile        |
| Conflicting human approvals | Human decisions disagree           | Escalate                             |

## 26. Abuse and Misconfiguration Risks

### 26.1 Overbroad Exceptions

Temporary exceptions may become hidden permanent permissions.

Mitigation:

* expiration required,
* max uses,
* exception audit,
* CHL exception-rate monitoring.

### 26.2 Policy Fragment Sprawl

Too many fragments can create inconsistent behavior.

Mitigation:

* policy registry,
* conflict proofs,
* periodic constitutional review.

### 26.3 Human Override Misuse

Humans may approve actions that violate higher policies.

Mitigation:

* non-exceptionable policy flag,
* higher-authority requirement,
* attestation and audit.

### 26.4 Tool Policy Bypass

Agents may route around restricted tools.

Mitigation:

* tool calls require PCL query,
* side-effect class enforcement,
* execution record logging.

### 26.5 Policy Compiler Bugs

Wrong composition can allow forbidden actions.

Mitigation:

* unit tests,
* replay tests,
* deterministic hashing,
* independent audit of critical policies.

### 26.6 Silent Policy Mutation

A layer may change policy without recording it.

Mitigation:

* policy fragment versioning,
* attestation hash,
* policy ledger.

## 28. Reference Pseudocode

```python id="cfw6s0"
def is_action_allowed(context, action):
    query = PolicyQuery(
        context=context,
        action=action,
        requested_by=context.agent_class or "unknown",
    )

    fragments = load_policy_fragments(context)
    active_fragments = filter_non_expired(fragments)

    scoped_rules = []
    for fragment in active_fragments:
        if scope_matches(fragment.scope, context, action):
            for rule in fragment.rules:
                if rule_matches(rule, context, action):
                    scoped_rules.append(rule)

    active_exceptions = load_active_exceptions(context, action)

    conflicts = detect_conflicts(scoped_rules, active_exceptions)

    resolutions = []
    for conflict in conflicts:
        resolution = resolve_conflict(conflict)
        resolutions.append(resolution)

        if resolution.method == "unresolved_block":
            attestation = create_attestation(
                context=context,
                policy_fragments=active_fragments,
                matched_rules=scoped_rules,
                conflicts=conflicts,
                exceptions=active_exceptions,
            )

            return AllowanceResult(
                query_id=query.query_id,
                decision="block_due_to_conflict",
                allowed=False,
                reason="Unresolved policy conflict.",
                constraints=[],
                matched_rules=[r.rule_id for r in scoped_rules],
                conflict_proof_ids=[c.conflict_id for c in conflicts],
                exception_required=False,
                human_approval_required=True,
                dependency_note_required=False,
                policy_attestation_id=attestation.attestation_id,
            )

    decision = compute_final_decision(scoped_rules, resolutions, active_exceptions)

    attestation = create_attestation(
        context=context,
        policy_fragments=active_fragments,
        matched_rules=scoped_rules,
        conflicts=conflicts,
        exceptions=active_exceptions,
    )

    return AllowanceResult(
        query_id=query.query_id,
        decision=decision.kind,
        allowed=decision.allowed,
        reason=decision.reason,
        constraints=decision.constraints,
        matched_rules=[r.rule_id for r in scoped_rules],
        denied_by=decision.denied_by,
        conflict_proof_ids=[c.conflict_id for c in conflicts],
        exception_required=decision.exception_required,
        human_approval_required=decision.human_approval_required,
        dependency_note_required=decision.dependency_note_required,
        policy_attestation_id=attestation.attestation_id,
    )
```

## 32. Summary

PCL-003 is the policy spine of the stack.

It ensures that multi-agent systems do not depend on whichever instruction happens to be closest in context.

It composes policies deterministically, resolves conflicts, manages exceptions, and produces reproducible policy attestations.

PCL does not decide what is useful.

It decides what is allowed.
