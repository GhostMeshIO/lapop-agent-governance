## 1. Purpose

EIL-003 defines the Economic Intelligence Layer for multi-agent LLM orchestration.

It governs cost-aware, value-aware, uncertainty-aware decisions during execution.

EIL answers the question:

> Is another action worth taking?

In multi-agent systems, every additional step has cost:

* token cost,
* time cost,
* tool cost,
* human review cost,
* contamination risk,
* merge complexity,
* coordination overhead,
* opportunity cost,
* and possible downstream regret.

EIL exists because agent systems often overcompute, over-audit, over-retry, over-escalate, or continue expanding a task long after marginal value has collapsed.

The core principle:

> Do more work only when expected value exceeds expected cost and risk.

---

## 2. Design Goals

EIL-003 aims to make execution decisions:

* value-aware,
* cost-aware,
* uncertainty-aware,
* latency-aware,
* human-attention-aware,
* tool-trust-aware,
* risk-sensitive,
* and auditable.

EIL should help decide when to:

* retrieve more context,
* run another audit,
* call a tool,
* ask a human,
* run another model,
* repair again,
* stop,
* defer,
* preserve disagreement,
* or accept the current result.

---

## 3. Non-Goals

EIL-003 does **not**:

* override policy,
* validate artifacts directly,
* decide truth,
* merge outputs,
* execute tools directly,
* update strategy tables directly,
* maintain trust graphs,
* perform rollback,
* or mutate artifacts.

EIL recommends actions.

PCL decides whether actions are allowed.

LAPOP executes approved actions.

SLL may learn from EIL outcomes.

CHL may use EIL signals to understand collaboration burden.

TGL may use EIL signals to evaluate stale or expensive branches.

| Human Review Load Modeling      | Treat human attention as scarce and costly.                      |

| Repair Economics                | Decide whether another repair attempt is likely to pay off.      |

## 5. Core Concepts

### 5.1 Value of Information

Value of information is the expected benefit of taking an action that reduces uncertainty or improves quality.

Examples:

* retrieving one more document,
* running a test,
* running a second model,
* asking a human reviewer,
* running a deeper audit,
* delaying merge until a dependency resolves.

### 5.2 Expected Net Value

EIL ranks actions by expected net value:

```text
Expected_Net_Value =
Expected_Quality_Gain
+ Expected_Risk_Reduction
+ Expected_Rework_Avoidance
- Expected_Cost
- Expected_Latency
- Expected_Contamination_Risk
- Expected_Coordination_Burden
```

### 5.3 Human Attention Scarcity

Human review is not free.

It has:

* latency,
* cognitive cost,
* opportunity cost,
* fatigue,
* limited availability,
* and escalation burden.

EIL should avoid escalating low-value issues to humans.

### 5.4 Tool Trust

Tools differ by domain and task.

A tool may be reliable for syntax validation but weak for semantic judgment.

EIL tracks tool reliability by:

* domain,
* artifact type,
* task type,
* failure rate,
* reversal rate,
* latency,
* and cost.

### 5.5 Economic Regret

Economic regret occurs when a decision later appears wasteful or insufficient.

Examples:

* skipped audit caused later rollback,
* extra repair attempt did not improve result,
* human review was unnecessary,
* tool call produced no useful information,
* stopping early caused downstream failure.

## 8. Value Function

Recommended default:

```text
V(action) =
wq * expected_quality_gain
+ wu * expected_uncertainty_reduction
+ wr * expected_risk_reduction
+ ww * expected_rework_avoidance
+ wl * expected_learning_value
+ user_value
```

Where all values are normalized to `[0, 1]`.

Example:

```python
def estimate_value(action, uncertainty, quality, risk):
    return (
        0.30 * expected_quality_gain(action, quality)
        + 0.20 * expected_uncertainty_reduction(action, uncertainty)
        + 0.25 * expected_risk_reduction(action, risk)
        + 0.15 * expected_rework_avoidance(action, risk)
        + 0.10 * expected_learning_value(action)
    )
```

## 9. Cost Function

Recommended default:

```text
C(action) =
wc_token * token_cost
+ wc_tool * tool_cost
+ wc_latency * latency
+ wc_human * human_cost
+ wc_coord * coordination_cost
+ wc_opportunity * opportunity_cost
```

Example:

```python
def estimate_cost(action, budget, human_load):
    return (
        0.20 * token_cost_score(action, budget)
        + 0.15 * tool_cost_score(action, budget)
        + 0.20 * latency_score(action, budget)
        + 0.25 * human_cost_score(action, human_load)
        + 0.10 * coordination_cost_score(action)
        + 0.10 * opportunity_cost_score(action)
    )
```

## 10. Risk Penalty

Recommended default:

```text
R(action) =
rc * contamination_risk
+ rp * policy_risk
+ rb * boundary_risk
+ rm * merge_risk
+ rr * rollback_risk
+ rs * security_risk
+ rpriv * privacy_risk
```

Example:

```python
def estimate_risk_penalty(action, risk):
    return (
        0.25 * risk.contamination_risk
        + 0.20 * risk.policy_risk
        + 0.15 * risk.boundary_risk
        + 0.10 * risk.merge_risk
        + 0.10 * risk.rollback_risk
        + 0.10 * risk.security_risk
        + 0.10 * risk.privacy_risk
    )
```

## 11. Risk-Adjusted Score

```text
Score(action) =
Value(action)
- Cost(action)
- RiskPenalty(action)
+ TrustAdjustment(action)
- HumanLoadPenalty(action)
- LatencyPenalty(action)
```

Example:

```python
def risk_adjusted_score(action, value, cost, risk, trust, human_load, latency):
    return (
        value.total_value_score
        - cost.total_cost_score
        - risk
        + trust.adjustment
        - human_load.penalty
        - latency.penalty
    )
```

Highest score wins unless PCL denies the action.

## 12. Stop / Continue Rules

EIL should recommend `stop` when:

```yaml
StopConditions:
  - current_quality_score >= target_quality_score
  - uncertainty_score <= uncertainty_tolerance
  - expected_net_value_of_best_action <= 0
  - repair_attempts_used >= max_repair_attempts
  - token_budget_used >= token_budget_total
  - latency_used >= latency_budget_seconds
  - contamination_risk_too_high
  - policy_risk_too_high
```

EIL should recommend `continue` when:

```yaml
ContinueConditions:
  - expected_net_value_of_best_action > 0
  - critical uncertainty remains
  - validation failure is likely repairable
  - audit would substantially reduce risk
  - human review likely prevents major regret
  - dependency resolution unblocks high-value merge
```

## 13. Human Review Economics

### 13.1 When Human Review Is Worth It

Recommend human review when:

* expected harm or regret is high,
* policy ambiguity cannot be resolved,
* semantic change is high,
* safety risk is high,
* merge conflict is consequential,
* model confidence is low but stakes are high,
* human reversal probability is high.

### 13.2 When Human Review Is Not Worth It

Avoid human review when:

* issue is low-stakes,
* validation can resolve it,
* cost is high and value is low,
* human queue is overloaded,
* uncertainty is acceptable,
* action is advisory only,
* artifact can be preserved as disagreement.

### 13.3 Human Review Penalty

```python
def human_load_penalty(review_load):
    if review_load.availability_status == "unavailable":
        return 1.0
    base = min(1.0, review_load.current_queue_size / 20.0)
    fatigue = review_load.fatigue_risk_score or 0.0
    return 0.7 * base + 0.3 * fatigue
```

## 14. Tool Trust Calibration

Tools should be trusted by context.

Example:

| Tool             | Reliable For               | Less Reliable For           |
| ---------------- | -------------------------- | --------------------------- |
| Unit test runner | code execution correctness | semantic architecture       |
| Linter           | style/syntax               | business logic              |
| Type checker     | interface consistency      | factual correctness         |
| Web search       | current facts              | private project files       |
| LLM critic       | broad critique             | final factual authority     |
| Human reviewer   | high-stakes judgment       | cheap repetitive validation |

### Tool Trust Adjustment

```python
def tool_trust_adjustment(tool_record):
    return (
        0.5 * tool_record.reliability_score
        + 0.3 * tool_record.calibration_score
        - 0.2 * (tool_record.reversal_rate or 0.0)
    )
```

## 15. Repair Economics

Another repair attempt is worth it when:

* failure is specific,
* prior repair count is low,
* similar repairs succeed historically,
* validation can test the fix,
* risk of worsening artifact is low,
* expected cost is acceptable.

Stop repair when:

* repeated failures,
* error is ambiguous,
* repair introduces new errors,
* confidence decreases,
* semantic drift increases,
* policy or boundary issue remains unresolved.

Recommended rule:

```python
if repair_attempts >= max_repair_attempts:
    recommend_stop_or_escalate()

if repair_attempts >= 2 and validation_error_types_unchanged:
    recommend_alternate_agent_or_supervisor()
```

## 16. Audit Economics

Run another audit when:

* merge risk is high,
* artifact is high-impact,
* semantic change is high,
* evidence grounding is weak,
* contamination risk is moderate/high,
* previous audit findings are unresolved.

Avoid another audit when:

* recent audit passed,
* artifact is low-impact,
* expected new findings are low,
* audit cost is high,
* uncertainty is acceptable.

---

## 17. Retrieval Economics

Retrieve more when:

* evidence gap is high,
* current facts may be outdated,
* citations are required,
* high-stakes accuracy matters,
* private project context is likely missing,
* unresolved ambiguity affects merge.

Avoid retrieval when:

* user provided all needed content,
* task is rewriting or formatting,
* retrieval source is irrelevant,
* latency budget is exhausted,
* expected information gain is low.

---

## 18. Disagreement Economics

Preserving disagreement can be economically rational.

Preserve disagreement when:

* merge would hide uncertainty,
* two outputs are both plausible,
* evidence is insufficient,
* policy conflict is unresolved,
* human review is too costly now,
* future information may resolve it.

Do not preserve disagreement when:

* one side is clearly invalid,
* disagreement blocks urgent low-risk work,
* policy requires a decision,
* ambiguity is artificial.

## 19. Budget Enforcement

EIL should track budgets continuously.

### Budget States

```yaml
BudgetStatus:
  - under_budget
  - near_budget
  - over_budget
  - budget_unknown
```

### Near-Budget Behavior

When near budget:

* raise stop threshold,
* prefer low-cost validation,
* avoid speculative expansion,
* reduce retries,
* preserve disagreement instead of resolving low-value conflicts.

### Over-Budget Behavior

When over budget:

* stop unless high-risk issue remains,
* require supervisor approval,
* avoid human escalation unless critical,
* write partial result if useful.

## 28. Abuse and Misconfiguration Risks

### 28.1 Under-Auditing High-Stakes Work

EIL may stop early to save cost.

Mitigation:

* high-risk tasks raise audit value,
* safety policy constraints from PCL,
* regret tracking.

### 28.2 Over-Optimizing for Cheapness

Cheap outputs may be low quality.

Mitigation:

* quality floor,
* risk floor,
* validation requirements,
* human reversal monitoring.

### 28.3 Human Review Avoidance

The system may avoid human review because it is expensive.

Mitigation:

* high-stakes escalation triggers,
* policy-required review,
* regret records for missed review.

### 28.4 Tool Trust Drift

Tool reliability may change over time.

Mitigation:

* decaying calibration,
* periodic recalibration,
* domain-specific trust records.

### 28.5 Coordination Cost Explosion

Agent systems may spend too much effort coordinating.

Mitigation:

* note-density cost,
* boundary churn cost,
* stop/defer recommendations.

## 30. Reference Pseudocode

```python
def decide_next_action(request):
    scored_actions = []

    for action in request.candidate_actions:
        policy_result = pcl.is_action_allowed(
            context=request.context,
            action=action.action_descriptor,
        )

        if not policy_result.allowed:
            continue

        cost = estimate_cost(action, request.budget_state)
        value = estimate_value(
            action,
            request.uncertainty_state,
            request.quality_state,
            request.risk_state,
        )
        risk_penalty = estimate_risk_penalty(action, request.risk_state)
        trust_adjustment = estimate_trust_adjustment(action)
        human_penalty = estimate_human_load_penalty(action)
        latency_penalty = estimate_latency_penalty(action, request.budget_state)

        final_score = (
            value.total_value_score
            - cost.total_cost_score
            - risk_penalty
            + trust_adjustment
            - human_penalty
            - latency_penalty
        )

        scored_actions.append((action, final_score, cost, value))

    if not scored_actions:
        return make_decision_record(
            decision="stop",
            rationale="No allowed positive-value actions available.",
        )

    scored_actions.sort(key=lambda x: x[1], reverse=True)
    best_action, best_score, best_cost, best_value = scored_actions[0]

    if best_score <= 0:
        return make_decision_record(
            decision="stop",
            recommended_action=best_action,
            expected_net_value=best_score,
            rationale="Best action has non-positive expected net value.",
        )

    return make_decision_record(
        decision="continue",
        recommended_action=best_action,
        expected_net_value=best_score,
        rationale="Best action has positive expected net value.",
    )
```

## 34. Summary

EIL-003 is the cost/value brain of the stack.

It prevents agent systems from blindly doing more work.

It treats computation, tools, time, human attention, coordination, and risk as scarce resources.

EIL does not decide what is allowed.

PCL does that.

EIL does not decide what is true.

Validation, audit, evidence, and humans do that.

EIL decides whether another action is worth it.
